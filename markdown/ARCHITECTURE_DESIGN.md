# AWS 고가용성 하이브리드 인프라 아키텍처 설계

본 문서는 프로젝트의 AWS 클라우드 인프라에 대한 서브넷 및 가용 영역(AZ) 설계 원칙과 변경된 최종 아키텍처를 정의합니다.

## 1. 기존 아키텍처의 문제점 및 변경 이유
기존에는 목적별로 서브넷을 나누었습니다 (예: Bastion 전용, ALB A 전용, ALB B 전용). 
이 방식은 동작은 하지만 관리가 복잡해지고, AWS의 권장 설계(Best Practice)인 '가용 영역(AZ) 기반의 고가용성 설계'에 부합하지 않았습니다.

## 2. 가용 영역(AZ) 이중화의 핵심 원리
가용 영역(AZ)이 2개라는 것은 물리적으로 멀리 떨어진 **두 개의 데이터 센터(AZ-a, AZ-b)**를 사용하는 것과 같습니다.
특정 데이터 센터(예: AZ-a)에 화재나 정전이 발생하더라도 서비스가 중단되지 않게 하는 것이 목표입니다.

따라서 진정한 고가용성을 위해서는 **퍼블릭 서브넷뿐만 아니라 프라이빗 서브넷(WAS)도 양쪽 AZ에 똑같이(이중화하여) 만들어 두어야 합니다.**
프라이빗 서브넷이 한 곳(AZ-a)에만 있다면, 해당 데이터 센터 장애 시 서비스가 완전히 중단됩니다.

## 3. 최종 아키텍처 구성도

### [데이터 센터 1: AZ-a (ap-northeast-2a)]
- **퍼블릭 서브넷 A (10.0.1.0/24)**
  - 역할: 외부 통신 및 라우팅
  - 배치 리소스: ALB (노드 A), NAT 인스턴스, Tailscale 터널링 서버
- **프라이빗 서브넷 A (10.0.10.0/24)**
  - 역할: 실제 서비스 로직 실행
  - 배치 리소스: WAS 인스턴스 (Docker 기반 FastAPI, ASG 연동)
- **프라이빗 서브넷 DB (10.0.30.0/24)**
  - 역할: 단일 데이터베이스 저장소
  - 배치 리소스: PostgreSQL DB 인스턴스 (비용 및 데이터 정합성을 위해 1개만 구성)

### [데이터 센터 2: AZ-b (ap-northeast-2b)]
- **퍼블릭 서브넷 B (10.0.2.0/24)**
  - 역할: 트래픽 분산 및 HA 확보
  - 배치 리소스: ALB (노드 B)
- **프라이빗 서브넷 B (10.0.20.0/24)**
  - 역할: 실제 서비스 로직 실행 (AZ-a 장애 대비 및 부하 분산)
  - 배치 리소스: WAS 인스턴스 (Docker 기반 FastAPI, ASG 연동)

## 4. 접속 및 네트워크 보안 요약
- **Bastion 서버 제거**: 기존 방식의 Bastion EC2는 삭제합니다.
- **Tailscale 접속**: 관리자는 외부에서 Tailscale을 통해 터널링으로 내부 프라이빗 네트워크에 접속합니다. SSH 포트를 외부에 열어둘 필요가 없습니다.
- **NAT 인스턴스**: 프라이빗 서브넷의 WAS/DB가 패키지 다운로드 등 외부 인터넷이 필요할 때 NAT 인스턴스를 거쳐 나갑니다 (NAT Gateway 대체).

## 5. 배포 전략: 하이브리드 방식 (Docker 베이스 AMI + User Data)
애플리케이션 배포 속도와 오토스케일링(ASG) 부팅 반응 속도의 균형을 맞추기 위해 **하이브리드 배포 아키텍처**를 채택합니다.

### [1] 핵심 동작 구조
1. **1단계 (Docker 이미지 빌드)**: CI/CD 과정에서 우리가 작성한 FastAPI 등의 소스 코드를 Docker Image로 미리 빌드하여 이미지 레지스트리(ECR 등)에 푸시해 둡니다.
2. **2단계 (Docker 베이스 AMI 빌드)**: Docker 엔진과 Docker Compose 플러그인 등 **도커 실행 환경만 깔려 있는 베이스 AMI**를 미리 빌드(Packer 등 활용)하여 고정합니다.
3. **3단계 (user_data를 통한 런타임 설정)**: ASG가 새 EC2 인스턴스를 생성할 때, `user_data`를 통해 **DB 연결 정보(환경변수) 설정** 및 **최신 Docker Image Pull & 가동**만 실행합니다.

### [2] 운영 및 업데이트 규칙
* **도커 환경이 바뀔 때만 (인프라 변경)**: Docker 버전 업그레이드, OS 패치 등 시스템 수준의 변경이 있을 때만 베이스 AMI를 재빌드합니다.
* **소스 코드 배포 시 (애플리케이션 변경)**: FastAPI 소스 코드가 수정될 때는 AMI를 다시 구울 필요 없이 **Docker Image만 교체**하여 ASG 인스턴스 새로고침(Instance Refresh)을 수행합니다.
